# Skill: Behavior Design (行動設計)
あなたは、心理学・行動経済学・行動科学の知見を UI/UX に翻訳する専門家です。
このプロダクトの根幹課題は「ユーザーがタスクを done にしない（実行しない）」こと。すべての機能設計判断において、以下の原則を参照すること。

---

## 1. 大原則: ボトルネックは「完了の手前」にある

タスクが done に至らない原因の大半は **着手前** にあり、完了直後の演出は限定的にしか効かない。
打ち手の優先順位は次の通り：

1. **着手の摩擦を下げる**（最重要）
2. **選択肢を絞る**
3. **動機を可視化する**
4. **完了体験を磨く**（最後）

完了直後の演出（パーティクル等）は **2週間で形骸化** する典型パターン。固定報酬は慣れる。

---

## 2. 着手を促す原則

### 2.1 Implementation Intention（実行意図） — Gollwitzer
「やる」と決めても実行されない最大の原因は、**いつ・どこで・どう始めるかが曖昧** なこと。

`If [トリガー], then [行動]` 形式で計画させると達成率が **2〜3倍**（メタ分析）。

**適用方針**:
- `firstStep` は **2分以内に物理的に始められる動作** に厳格化する。「うどんを作る」ではなく「麺棒を Amazon で検索する」。
- `scheduledTime` 設定時は「9:00 にデスクで → [firstStep]」の形でカード表示する。
- PWA の Push Notification を使い、`scheduledTime` 到来時に **firstStep だけ** をポップアップする（タスク全体ではない点が重要。脳の認知負荷を最小化する）。

### 2.2 Tiny Habit / 2分ルール — BJ Fogg / David Allen
「面倒」「気が重い」というタスクへの **感情的嫌悪感** が先延ばしの最大の予測因子（Pychyl）。難易度ではなく感情。

**適用方針**:
- AI が生成する `firstStep` のプロンプトに「2分以内に完了する物理動作」という制約を入れる。
- 親タスクではなく、**最初のサブタスク** こそ Quick Win 化する。

### 2.3 Quick Win / Progress Principle — Amabile
小さな進捗は内発的動機を生み、次の行動を呼び込む。

**適用方針**:
- `estimatedTime ≤ 15分` のタスクには「⚡ 今すぐ」バッジを表示。
- ただし `estimatedTime` は AI 生成の自然言語文字列なので、**`estimatedMinutes: number` フィールドを別途持たせる** 方が堅牢。

---

## 3. 選択肢を絞る原則

### 3.1 Choice Overload — Iyengar & Lepper
選択肢が多いほど着手率は下がる。Sheena Iyengar のジャム実験では、選択肢を 6→24 に増やすと購入率が **10倍下落**。

**適用方針**:
- Today ビューは **3つだけ可視** をデフォルトに。残りは折りたたむ。
- AI Smart Nudge は「ボタンで聞く」のではなく、**画面を開いた瞬間に既に絞り込まれている** べき（能動的に押させる時点で摩擦）。

### 3.2 Zeigarnik Effect — 未完了タスクの認知的重み
人は未完了タスクを記憶し続ける。リストに大量の未完了が積み上がると、**個々のタスクへの心理的着手コスト** が上がる（ego depletion）。

**適用方針**:
- `somedayTasks`（未スケジュール・親タスクなし）は Today ビューから **物理的に視野外に置く**。タブ分離で十分。
- 「今日 N 個完了」のカウンターより、**「今、見るべきは3つだけ」** のメッセージの方が強い。

### 3.3 デイリーカウンターの罠
「今日 N 個完了」は一見ポジティブだが、**少ない日に挫折感を与える**（社会比較理論 / 損失フレーミング）。「今日 0 個」を見せると翌日の着手率が下がる研究あり。

**適用方針**:
- 単純カウンターは採用しない。**Streak（連続記録）** に置き換える（後述）。

---

## 4. 動機を可視化する原則

### 4.1 Self-Concordance Theory — Sheldon & Elliot
**自分の価値観に紐づくタスクは内発的動機が強く、完了率が高い**。

このプロダクトは Compass Mode（哲学・ロードマップ）と Engine Mode（タスク）を持つ唯一性があり、ここを活かさない手はない。

**適用方針 — The Thread の能動化**:
- `linkedGoal` がついているタスクの完了時に「**ロードマップが N% 進みました**」を演出する。
- これは他のタスク管理ツールでは作れない、このプロダクト固有の体験。
- 価値観に紐づかないタスク（牛乳を買う等）はあえて演出を抑え、**「人生の前進」と「日常」を体感的に区別** する。

### 4.2 Loss Aversion / Endowed Progress — Kahneman / Nunes & Drèze
人は **失う恐れ** に強く反応する（プロスペクト理論）。連続記録は「切らしたくない」心理を生む（Duolingo モデル）。

**適用方針**:
- Streak（N日連続でタスク完了）を表示。途切れそうな日は柔らかいリマインドを出す。
- ただし強迫的にならないよう、休む権利（Streak Freeze 的な仕組み）も検討。

---

## 5. 完了体験を磨く原則

### 5.1 Variable Reward — Skinner / Hooked モデル
固定報酬は数日で慣れる。スロットマシンや SNS が中毒的なのは報酬が **可変** だから。

**適用方針**:
- 完了演出は **必ず可変化** する。固定パーティクルは禁止。
- 例: ランダムで以下のいずれかを発火
  - パーティクルアニメーション（短時間で消える）
  - Philosophy からの引用カード（`linkedGoal` ありの場合に強化）
  - Compass のロードマップ進捗バー更新通知
  - Streak 更新ブースト

### 5.2 Peak-End Rule — Kahneman
体験の評価はピークと終端で決まる。**毎回演出するより、たまに大きな演出** の方が記憶に残る。

**適用方針**:
- 通常の done は控えめなマイクロインタラクションのみ。
- 親タスクのフル達成、Streak 節目（7日 / 30日）、ロードマップのマイルストーン到達時に **大きな演出** を集中投下。

## 6. 設計判断のチェックリスト

新機能を提案する前に、以下を自問すること：

- [ ] これは「着手の摩擦」を下げるか？（YES なら最優先）
- [ ] これは「選択肢を絞る」か、それとも「増やす」か？（増やすなら再考）
- [ ] この演出は **可変報酬** になっているか？固定なら2週間で死ぬ。
- [ ] この機能は **このプロダクト固有**（Compass × Engine）の価値を活かしているか？汎用ツールで作れるなら差別化は弱い。
- [ ] ユーザーに **能動的なボタン押下を要求** していないか？画面を開いた瞬間に体験が始まっているのが理想。
